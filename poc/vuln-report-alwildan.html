Laporan Kerentanan: IDOR dan Privilege Escalation di Sistem WISE Al-Wildan Islamic School Bekasi

Analisis profesional oleh Abdurrohman Al Fathi mengenai kerentanan kritis di sistem manajemen karyawan WISE Al-Wildan Islamic School Bekasi pada halaman https://wise.alwildanbekasi.sch.id/login-employee/profile. Laporan ini mencakup metodologi penemuan, proof of concept, dampak, dan rekomendasi perbaikan untuk tujuan edukasi dan peningkatan keamanan siber.

Pendahuluan

Laporan ini menyajikan temuan kerentanan Insecure Direct Object Reference (IDOR) dan Privilege Escalation pada sistem WISE (AlWildan Integrated Smart Education) di Al-Wildan Islamic School Bekasi, khususnya pada halaman profil karyawan https://wise.alwildanbekasi.sch.id/login-employee/profile.

Sistem WISE digunakan untuk manajemen profil karyawan, termasuk pengubahan data pribadi dan akun. Temuan ini menunjukkan bagaimana pengguna biasa dapat memanipulasi parameter untuk mengakses dan mengubah akun pengguna lain, termasuk role admin IT, yang berpotensi menyebabkan pengambilalihan akun dan akses tidak sah.

Metodologi Penemuan

Apa: Proses sistematis untuk mengidentifikasi kerentanan.

Mengapa: Memastikan penemuan berdasarkan analisis logic, bukan keberuntungan.

Sebagai pentester dengan pengalaman CTF dan hardening, saya mulai dengan inspeksi sumber kode halaman profil. Fokus pada form data pribadi, di mana field NIP tampak 'disabled' tapi potensial untuk manipulasi.

  1. Reconnaissance: Analisis HTML menggunakan dev tools. Identifikasi field NIP di div data pribadi.
  2. Hipotesis: Jika server tidak validasi NIP terhadap session user, manipulasi bisa sebabkan IDOR.
  3. Testing: Modifikasi klien-side untuk ubah NIP ke target (e.g., IT support).
  4. Escalation: Gabung dengan ubah password untuk takeover.

Proses ini tunjukkan pemahaman mendalam: Kontrol klien rapuh; security di server-side krusial.

Analisis Kerentanan

Nama Kerentanan: Insecure Direct Object Reference (IDOR) dengan Insufficient Authorization, sebab Privilege Escalation.

Penyebab: Field NIP 'disabled' di klien, tapi nilai dikirim ke server tanpa verifikasi ownership.

Potongan Kode Relevan: Berikut div form yang rentan:

<div class="mb-3">
    <label class="form-label required">Nama</label>
    <input type="text" class="form-control form-control-sm" name="nama" value="Fikriyyah Hanifah " required>
</div>
<div class="mb-3">
    <label class="form-label required">NIP</label>
    <input type="text" class="form-control form-control-sm" name="nip" value="020625146" disabled>
</div>
<div class="mb-3">
    <label class="form-label required">Jabatan</label>
    <input type="text" class="form-control form-control-sm" name="jabatan" value="Guru" disabled>
</div>

Penjelasan: Value NIP bisa diubah via dev tools (hapus 'disabled', ganti ke NIP target seperti IT support). Saat submit, server proses tanpa cek, izinkan ubah data/password. Ini IDOR karena referensi langsung (NIP) tanpa autorisasi.

Backend PHP custom, vuln dari kurang validasi server-side.

Proof of Concept (PoC)

Langkah-langkah PoC: (Dilakukan di test env; jangan coba produksi tanpa izin.)

  1. Login sebagai user biasa di https://wise.alwildanbekasi.sch.id/login-employee/.
  2. Buka https://wise.alwildanbekasi.sch.id/login-employee/profile, tab 'Data' atau 'Akun'.
  3. Dev tools: Di div NIP, hapus 'disabled', ubah value ke NIP IT support (dari recon).
  4. Ubah password di form, submit.
  5. Logout, login dengan NIP IT dan password baru. Akses admin berhasil.

Catatan: Ini sebab account takeover karena NIP=username untuk login.

Contoh screenshot PoC: Ubah value NIP untuk takeover.

Dampak dan Risiko

Tingkat Keparahan: High (CVSS ~8.1: AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H).

  • Account Takeover: Akses role tinggi, ubah data sekolah.
  • Data Leak: Profil, dokumen, info siswa bocor.
  • Escalation: IT role punya akses luas, potensi sabotase.
  • Reputasi: Kebocoran data rusak kepercayaan sekolah.

Rekomendasi Mitigasi

  • Server Validation: Cek NIP submit == session NIP.
  • RBAC: Batasi edit ke user sendiri.
  • Tokens: Gunakan CSRF dan hash NIP hidden.
  • Logs: Catat perubahan untuk audit.
  • Audit: Pentest reguler oleh expert seperti Abdurrohman Al Fathi.

Ikuti OWASP guidelines untuk web security.
